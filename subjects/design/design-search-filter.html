搜索与筛选

🎮 交互 · 难度 ●●○ · 落在后台列表、表格的搜索框与筛选器

后台系统里几乎每个列表页都有同一个难题:一张表里几百上千条数据,用户怎么快速找到他要的那几条。解决它的工具就三样——搜索框、筛选器、排序。听起来简单,但做后台/管理系统时,这三样最容易做歪:搜索框藏在角落、筛了之后看不出自己筛了什么、一刷新条件全没了。这一页把它们各自的分工、设计要点和常见坑讲清楚,例子全落在你天天看的「客户列表、订单表格」上。

🔑 一句话先抓住分工

搜索=用户脑子里已经有个关键词,想直接命中它(某个客户名、某个编号);筛选=用户不知道具体是哪条,只知道按某个维度收窄(只看华东的、只看活跃的);排序=不改变有哪些,只调先后顺序(按金额从高到低)。三件事经常被混在一起做,结果谁都没做好。先把它们拆开,再组合。

1. 搜索、筛选、排序,别混成一团

这三样解决的是不同的问题,放错位置用户就会别扭。

  • 搜索——自由输入,模糊匹配。用户心里有目标:"我要找那个叫『王海』的客户"。给一个输入框,边打边找。
  • 筛选——按预设维度收窄。用户没有具体目标,只有范围:"给我看华东、状态是活跃的客户"。给一组可选的维度(地区、状态、标签),不是让他打字。
  • 排序——只换顺序,不删条目。"按成交金额从高到低排一下"。点表头或选下拉,列表里的条数一条不变。

一个典型的 NocoBase 表格区块,这三样会同时出现:表格上方一个搜索框,旁边几个筛选 chips,表头点一下能升降序。它们是叠加关系——可以先搜「王」,再筛「华东」,再按金额排——而不是互相覆盖。这个"叠加"很关键,后面第 4 节会专门讲它最容易在哪里翻车。

2. 搜索框:让人一眼看到、打了字就有反应

搜索框是用户最先去找的东西,所以它的位置和反馈最不能含糊。

位置要显眼

搜索框放在列表/表格正上方,占一整行或半行宽,别缩成一个小图标藏进工具栏菜单里。后台用户一天要搜几十次,每次都先点开菜单再搜,会被烦死。

即时搜索 vs 回车触发,看数据量选

"边输入边出结果"(search-as-you-type)体验最顺,用户打一个字列表就收窄一截,很快就能看出该不该继续打。但它不是无脑用的:

  • 数据量小、前端能一次拿全(比如几百条已经在浏览器里):放心用即时搜索,在前端过滤,零延迟。
  • 数据量大、每次搜索都要打后端接口:即时搜索会把请求打爆。这时要么加防抖(下面讲),要么干脆改成回车 / 点按钮才查,给用户一个明确的"我搜了"的动作。

即时搜索必须防抖

"防抖"就是:用户在打字,先别急着查,等他停下来 200~300 毫秒再发起一次查询。否则输入「客户」两个字,会在「客」「客户」之间各查一次,白白多打一次后端,结果还可能因为先后返回顺序错乱而闪烁。本页底部的实验台就做了防抖,你可以连续快打感受一下。

占位提示要写清能搜什么

✅ 好的占位提示

placeholder 写 搜索客户名 / 电话 / 编号。用户一看就知道这个框认哪些字段,不会拿订单号去搜一个只认名字的框。

⚠️ 偷懒的占位提示

placeholder 只写 搜索 或 请输入。用户得自己猜:能搜名字吗?能搜手机号吗?支持模糊吗?猜错一次就对这个框失去信任,下次宁可手动一页页翻。

空结果要给出路,不要只留白

搜不到东西时,最差的做法是显示一片空白——用户分不清是"真没有"还是"页面挂了"。好的空结果会说人话 + 给下一步:"没找到包含『王海明』的客户。试试只搜一部分(比如『王海』),或检查有没有多打了字。"

3. 筛选器:外露常用维度,且筛了要看得见

常用的外露,次要的收起来

不要把十几个筛选维度全平铺出来,那等于没有重点。把用得最多的 2~3 个维度做成一排可点的标签(chips)直接外露(比如客户列表的「地区」「状态」),其余次要的收进一个「更多筛选」的弹层或抽屉。用户 80% 的需求靠外露那几个就解决了。

多选还是单选,看维度性质

  • 互斥的用单选:一条记录的「状态」只能是活跃或流失之一,筛选时选了「活跃」就不该还能同时选「流失」(那等于没筛)。做成单选 chips,点一个亮一个。
  • 可叠加的用多选:「标签」一条记录可以同时有「大客户」「待回访」,用户也可能想看"大客户 或 待回访"的并集,这种做成多选。

已应用的筛选必须可见、可清除

⚠️ 最常见的坑:筛完看不出筛了什么

用户点了几下,列表变短了,但页面上没有任何地方告诉他"当前正在按 华东 + 活跃 过滤"。过一会儿他自己都忘了,看到列表里没有某个客户,以为系统丢数据了,其实只是被自己的筛选挡住了。这会直接引发"数据不见了"的误报工单。

✅ 正确做法:filter chips + 清空

把当前生效的每个条件,在列表上方排成一行小标签(filter chips):地区:华东 ✕ 状态:活跃 ✕。每个能单独点 ✕ 撤掉,旁边再放一个「清空筛选」一键还原。用户随时能看到"我现在被哪些条件框着",也能一步步松开。

筛完实时告诉用户剩多少条

每次筛选后,在显眼处更新一个计数:显示 23 / 共 480 条。这个数字让用户判断"筛得够不够狠"——还剩两百条说明条件太松,得再加一刀;只剩 0 条说明筛过头了,得松一点。没有这个数字,用户对自己离目标还有多远完全没数。

4. 组合与状态:筛选条件要留得住

搜索和筛选做对了单次体验,还有一层更隐蔽的体验:条件能不能保留。

别一筛选就丢了搜索词

用户先在搜索框打了「王」,再去点「华东」筛选——这两个条件应该同时生效(华东地区里名字带王的),而不是一点筛选就把搜索词清空、列表重新铺满。搜索和筛选是与(AND)的叠加,代码上要把它们当成同一组查询条件一起拼,而不是各管各的、互相覆盖。

刷新、返回、分享都不该丢状态

用户筛好了一屏,点进某条客户的详情看了看,返回时筛选条件最好还在,而不是回到一张白板从头再筛一次。更进一步,把筛选状态写进 URL 的 query 参数(?region=华东&status=活跃),这样刷新页面不丢,把链接发给同事,同事打开看到的是同一个筛选视图。后台系统里"把这个筛选结果发给老板看"是高频需求,URL 带状态能省掉大量截图和口头描述。

🔑 搜索筛选自查清单

下次做一个后台列表页,对着这几条过一遍:

☐ 搜索框在列表正上方、足够显眼,没藏进菜单
☐ placeholder 写清了能搜哪些字段,不是光秃秃一个"搜索"
☐ 即时搜索做了防抖(或大数据量改成回车触发)
☐ 搜不到时有"说人话 + 给下一步"的空结果提示
☐ 最常用的 2~3 个维度外露成 chips,次要的收进"更多筛选"
☐ 当前生效的筛选以 filter chips 显示,可单独 ✕、可一键清空
☐ 筛选后实时显示"X / 共 Y 条"
☐ 搜索词和筛选条件是叠加,不互相覆盖
☐ 刷新 / 返回 / 分享 URL 不丢筛选状态
☐ 大数据量查询时有加载态,空结果区分"无匹配"和"可能打错"

5. 性能与反馈:慢和空,都要给个交代

前面讲的是"该有哪些功能",这一节讲"当它慢下来、当它什么都搜不到,界面怎么不让人慌"。

大数据量要给加载态

当搜索/筛选要走后端、一时半会儿回不来,别让列表区一片死寂。给一个加载态(骨架屏或转圈),明确告诉用户"在查了,等一下"。没有这个反馈,用户会以为点击没生效,于是疯狂重复点击,反而把请求叠得更多。

空结果要分清两种"空"

✅ 区分对待两种空

• 条件太严,确实没有符合的:提示"当前筛选下没有客户,试试放宽地区或状态条件",并把"清空筛选"按钮放在旁边。
• 搜索词像是打错了:提示"没找到『王海明明』,检查一下有没有多打字,或只搜一部分关键词"。

这两种空给用户的下一步动作完全不同:一个该去松筛选,一个该去改输入。混成同一句"暂无数据"就等于把判断负担全甩给用户。

6. 试一试

✏️ 拿你自己的后台列表照一照

打开你正在做的某个 NocoBase 列表/表格页,对着回答三个问题:

1. 这张表里,用户最常用来收窄的是哪 2~3 个维度?(比如客户的「地区」「状态」、订单的「下单日期」「付款状态」)——这几个就该外露成 chips,而不是埋进"更多筛选"。

2. 现在筛选之后,页面上有没有一个地方显示"还剩多少条"?如果没有,用户怎么知道筛得够不够狠?

3. 筛完之后看得出当前生效了哪些条件吗?能单独撤掉某一个、能一键清空吗?如果不能,迟早会收到"数据怎么少了"的误报。

把三个答案写下来,差哪条就补哪条。多数后台列表,补上"维度外露 + 实时计数 + 可见可清的 filter chips"这三件,可用性就上一个台阶。

搜索负责"我知道找谁",筛选负责"我只知道范围",排序负责"换个顺序看"。把三者拆清楚、各自做到位,再让它们能叠加、能保留状态,一张几百上千条的后台表格就不再让人发怵了。
