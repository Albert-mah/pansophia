列表 · 详情 · 工作流

🗂️ 后台管理 · 难度 ●●○ · 更新于 2026-06-30

打开十个后台系统,里面的页面骨架都长得差不多:一张列表、一个详情、一套状态流转。这三样东西合在一起,几乎能搞定后台 80% 的场景——采购审批、工单处理、客户管理、库存出入库,本质上都是同一张骨架,只换了字段和业务规则。把这套骨架设计好,用户就算第一次见也能大致猜到怎么操作;设计坏了,再好的功能也会让人找不着北。

🧩 先有个具体画面

想象一个采购申请系统:财务小张需要采购一批服务器。她打开系统,先看到一张列表——里面列着所有采购单的名称、金额和当前状态。她点进自己那条,进入详情页——看到填写的内容、历史操作记录,然后点"提交审核"。单子流转到部门主管那里——主管在待办列表里找到它,点进去,看完之后点"通过"。状态从"待审核"变为"已通过",整个工作流走完。这个流程里的每一步,都依赖列表、详情、工作流三者协作。这一页就讲这套骨架怎么设计好。

一、列表 — 找得到、筛得快、批量省力

列表页的职责是帮用户快速定位目标记录。表格是最常见的呈现方式,但屏幕宽度有限,塞太多列用户要横向拖动才能看完,效率极低。挑列是列表设计里最常被忽视、却最影响体验的一步。

该露哪几列

好列表通常只露 5~7 列。选列原则是:

  • 标识符列(单据号 / 名称):让用户能定位"是哪一条"。
  • 核心业务值(金额 / 数量 / 联系人):用户扫一眼就能判断这条值不值得点开,通常 1~2 列够了。
  • 状态列:用颜色标签呈现,用户第一眼就想知道"这条现在到哪步了"。
  • 时间列(创建/更新时间):支持排序,让最新或最旧的记录能浮到顶部。
  • 操作列:把高频操作直接露出来(查看 / 编辑 / 删除),而不是藏进菜单。

不该出现在列表里的:描述/备注(太长,列里显示不完)、低频字段(内部编号、备用联系方式)、与列表主题无关的关联字段。这些进详情再看。

❌ 坏例:塞了 20 列

一张工单表有 28 个字段,列表页全部展示。用户要横向拖滚动条才能找到"状态"列,找到"操作"按钮又得再拖一次。重要信息淹没在不重要信息里——信息太全等于没有重点。

✅ 好例:精选 6 列

列表只露:工单号 / 标题 / 客户 / 状态 / 更新时间 / 操作。金额和描述进详情看。用户 3 秒内扫完一屏找到目标行,点"查看"直接进去。少即是多,用户不用思考就能用。

筛选条件的层次

列表通常需要筛选。筛选条件也有轻重之分:高频条件(如"状态""时间范围")直接摆在表格上方,用 tab 切换或下拉选择,点一下立刻生效;低频条件(如"申请部门""经手人")折叠进"高级筛选"面板里。两级设计让界面不拥挤,同时保留完整功能。

批量操作要按需浮出

用户勾选多行后,才会需要批量删除、批量导出、批量改状态。批量操作按钮不该一直挂在页面上——没有勾选时它没有意义,还占地方。正确做法:用户勾选行后,顶部实时浮出一条工具条,显示"已选 N 项"和可用操作。

✅ 好例:按需浮出批量工具条

表格默认状态:顶部只有搜索框和"新建"按钮。用户勾了 5 行后,顶部立刻浮出:"已选 5 项 · 批量导出 · 批量删除 · 取消选择"。需要时在那儿,不需要时消失——符合诺曼的"可见性"原则。

二、主从布局 vs 跳转详情页

详情页有两种打开方式:跳转(点一行导航到新页面)和主从布局(左边列表、右边实时显示详情,不离开这一页)。两者各有适用场景:

对比维度跳转详情页主从布局(左右分栏)
详情内容复杂程度详情内容多,需要全屏空间详情较精简,右侧够放
需要对比多条记录不方便(要来回跳转)方便(左边点,右边立刻换)
URL 可分享每条有独立 URL,可分享通常没有独立 URL
典型场景审批单、工单详情、合同邮件客户端、CRM 联系人列表

大多数管理后台用跳转更安全——详情内容往往比预期复杂得多;主从布局在移动端体验也会自动折叠成跳转。如果你的场景需要频繁在记录间切换对比,主从布局能显著提升效率。

三、详情 — 看清楚、能操作、留记录

详情页是用户决策的地方。打开一条采购申请,用户需要看清楚信息(申请了什么、多少钱、谁申请的),然后做出操作(通过、驳回、编辑),最后能追溯记录(谁在什么时间做了什么)。这三件事对应详情页的三个分区。

基本信息区 — 核心字段一屏看完

基本信息区放这条记录最核心的字段,通常以只读方式展示(或少量可编辑)。布局常用双列或三列网格——同类字段放一起(名称/金额/时间一组;申请人/部门/联系方式一组),视觉上容易扫读。字段太多时,可以把次要字段折叠进"展开更多"里。

关联数据区 — 附带展示相关子集

一条采购单可能关联多条附件、多行商品明细、多条历史版本。这些关联数据不应全塞进基本信息区——用 Tab 分页或折叠卡片分开放,用户按需展开。没有内容的 Tab 不显示,避免用户点进去看到空白。

操作日志区 — 所有操作有迹可查

这是最容易被遗漏的一块。谁创建了这条记录?谁提交了审核?谁在什么时间驳回的、驳回理由是什么?没有操作日志,事后追责无从查起,客服处理投诉也没有依据。所有关键操作(状态变更、编辑、附件上传)都应自动记录到日志里,带时间戳和操作人。

❌ 坏例:没有操作日志

采购单显示"已驳回",但详情里没有任何说明。财务问主管:你为什么驳回?主管说记得加了备注,但备注不知道去哪了。没有日志 = 没有记忆,所有争议只能靠人肉对账。

✅ 好例:完整的时间线

详情页底部有"操作记录"时间线:01-10 16:00 陈磊 · 创建申请 → 01-10 16:05 陈磊 · 提交审核 → 01-11 10:20 王总监 · 驳回(备注:请补充报价单)。任何时候回来都能还原当时的完整状态。

四、工作流 — 随时知道"在哪步、该谁做"

工作流让事情在人和人之间按规则流转——不靠口头传话、不靠微信催单。工作流设计最核心的一条原则:用户打开任意一条记录,一眼就能看出"现在在哪一步、当前该谁操作、他能做什么"。违反这条原则,工作流再完备也是一个黑盒。

步骤条:让进度一目了然

步骤条(Step Indicator)是工作流最直观的可视化方式:从左到右排列所有步骤,已完成的步骤打勾变色,当前步骤高亮,未来步骤灰显。用户看一眼就知道走到哪了、还剩多少步。步骤数不要超过 5 步——步骤太多会变成一行难以辨认的小点。如果流程确实复杂,考虑把多个小步骤归并成几个阶段展示。

状态标签的颜色约定

在列表和详情里,状态标签是用户第一眼聚焦的地方。颜色要有语义:灰色=草稿/中性;橙色=需要处理/等待;蓝色=进行中;绿色=正向完成;红色=负向结果/需注意。这套约定没有绝对标准,但在系统内选定后要全局统一——颜色混乱等于每次都要重新理解状态含义。

每步要明确写出:下一步该谁做

光知道"现在是待审核"还不够——是主管审核还是财务审核?他要去哪里操作?好的工作流设计会在状态旁边明确写出:"等待 王总监 审批",并且对应的人在他的待办列表里能直接看到这条任务。审批人不需要别人催,系统替你催。

❌ 坏例:状态是黑盒

状态显示"待审核"。提交人不知道谁在审、审到哪了,只能去问主管。主管说没收到通知,在系统里找了半天才找到这条。整个流程靠人工催单推动,系统形同虚设。

✅ 好例:状态透明,下步可见

详情页步骤条下方写清楚:"等待 技术部主管 王总监 审批",并显示提交时间。王总监登录后首页的"我的待办"里直接看到"1 条待你审批的采购申请"。提交人能看实时进度,不需要催人。

🔑 三件套自查清单

搭一个后台模块时,逐条问下面这些问题。任何一条答"不确定",就是需要补的设计:

列表

1. 列表只露了最关键的 5~7 列,没有塞全部字段吗?

2. 状态列清晰可见,用了颜色标签吗?

3. 高频筛选条件直接可见,低频的折叠了吗?

4. 批量操作只在勾选后才浮出,不占常驻空间吗?

详情

5. 详情页有"基本信息 / 关联数据 / 操作日志"三个分区吗?

6. 操作日志自动记录了所有关键操作(状态变更、编辑、提交)?

7. 当前可执行的操作按钮明确可见,不可操作的已禁用吗?

工作流

8. 步骤条一眼看出当前在第几步?步骤数不超过 5 个?

9. 当前状态旁边写明了"下一步该谁操作"吗?

10. 待办人在他的首页能看到"有 N 条待我处理"的提醒吗?

🎛️ 列表 → 详情 → 工作流,走一遍

五、试一试

✏️ 给自己系统里的一个对象套三件套框架

挑你正在做(或正在用)的后台里的一个对象(订单 / 工单 / 合同 / 库存物品),按三件套框架填空:

列表该展示哪几列?(尝试只选 5 列)

标识符列: ___  ·  核心值列: ___  ·  状态列: ___  ·  时间列: ___  ·  操作列: ___

详情页三个区各放什么?

基本信息: ___(3~5 个字段)  ·  关联数据: ___(1~2 个子集)  ·  操作日志: 自动记录哪些动作?

状态有哪几个?流转路径是?

草稿 → ___ → ___ → 完成 / 关闭。每次状态变更,该通知谁?

把上面三段填完,你的模块设计已经有 70% 了——剩下的 30% 是细节交互,但骨架已经立起来了。

列表、详情、工作流这套骨架的价值在于:它不是某个特定系统的做法,而是经过无数后台验证的通用模式。下次接手新模块、遇到"这个页面该怎么组织"的问题,先套这个骨架,再按业务细化——90% 的场景都够用。
