信息架构:导航 · 分类 · 层级

🎮 交互 · 难度 ●●○ · 更新于 2026-06-30

做后台系统,功能越堆越多,真正让人抓狂的往往不是某个功能不好用,而是「东西找不到」:这个操作到底在哪个菜单?为什么导出在「设置」里、新建却在「工具」里?把内容和功能组织成用户找得到的结构,这件事叫信息架构(Information Architecture,简称 IA)。这一页讲清三件互相咬合的事:分类(怎么分组)、层级(分几层、每层多宽)、导航(用什么形式把结构呈现出来),例子都落在后台的「左侧菜单、分组、面包屑、多级页面」上。

🧭 先有个画面感

把后台想成一栋大楼。分类是「哪些房间归在同一层」——财务的房间放一起、客服的房间放一起。层级是「大楼有几层、每层有几个门」——楼太高(深)爬得累,一层挤五十个门(宽)又眼花。导航是「楼里的指示牌」——左边的楼层索引(左侧菜单)、墙上的「您在 3 楼东侧」(面包屑)、房间内的分区标签(标签页)。

好的信息架构有一个朴素的检验标准:用户还没点开,就能猜到东西大概在哪。猜得到,说明你的分法和他脑子里的分法对上了;每次都要靠搜索或者挨个点开试,说明 IA 没做到位。

🔑 一句话抓住它

信息架构做三件事:① 分类——按用户怎么想把功能分组,不是按数据库表或公司部门;② 层级——在「太宽」和「太深」之间找平衡,后台一般 2~3 层最舒服;③ 导航——选对呈现形式(左侧菜单 / 顶部导航 / 面包屑 / 标签页),并随时告诉用户「你在哪、怎么回去」。三件事的共同目标只有一个:让人猜得到、找得着、不迷路。

1. 分类:按用户的心智,不按你的数据库

分类最常见的翻车,是照着系统内部的样子去分:数据库有几张表就分几个菜单、公司有几个部门就开几个入口。这对开发者很自然,对用户却是灾难——用户不知道你的表结构,也不关心你们部门怎么划分。他只带着一个任务来:「我要给这单退款」「我要导出这个月的报表」。分类要顺着用户的任务和心智来切。

按内部结构分(别这样):「主表」「子表」「关联数据」「日志表」——这是把数据库目录直接搬到了菜单上,用户一个都看不懂。

按用户任务分(这样):「客户」「订单」「商品」「报表」「设置」——每一项都对应用户脑子里一件要做的事,他扫一眼就知道该往哪去。

同样重要的是用用户的词,不用内部黑话。菜单叫「实体管理」「工单流转」「主数据维护」,用户得先翻译一遍才敢点;叫「客户」「订单」「商品库」,他一眼就懂。命名是分类的门面——分得再对,名字看不懂也等于没分。

🃏 卡片分类法(card sorting):让用户帮你分

不确定该怎么分组、怎么命名时,有个轻量又有效的办法叫卡片分类法:把每个功能 / 页面名字写在一张张卡片上,请几位真实用户按他们觉得合理的方式把卡片归堆,再给每堆起个名字。你会发现用户的分法常常和你设想的不一样——这正是值钱的地方。

两种玩法:开放式(用户自己分堆、自己命名,用来「从零摸清用户怎么想」)和封闭式(你先定好几个分类桶,用户只把卡片往桶里放,用来「验证你这套分类好不好用」)。后台改版前花半小时做一轮,能省掉后面无数次「这功能到底放哪」的争论。

2. 层级:广度和深度的拉锯

把功能分好组之后,马上要决定:分几层?每层放几项?这就是信息架构里最经典的取舍——广度(breadth)对深度(depth)。

  • 广度 = 每一层摆很多项、但层数少。极端就是全平铺:一个长长的列表,什么都在第一层。好处是点一下就到;坏处是几十项排成一墙,眼睛得逐行扫,扫起来比点还累。
  • 深度 = 每一层只摆几项、但层层往下钻。极端就是深嵌套:点开一级又一级,五六层才到目的地。好处是每屏很清爽;坏处是点到手酸,而且越深越容易迷路、忘了自己怎么进来的。

两边都难受,所以要找中间那个甜区。对大多数后台系统,经验值是不超过 2~3 层:顶层放能一眼扫完的几个大分组(通常控制在七八个以内),每个分组里再放该分组的具体功能。下面这个实验台,让你亲手感受这条权衡曲线。

3. 导航模式:什么时候用哪一种

结构定好了,还得选个呈现形式把它摆出来。后台常用四种,各有各的活,经常组合使用。

左侧菜单(后台主流)。纵向、能容纳多个分组和层级,是后台一级导航的默认选择。功能多、要分组、需要随时切换大模块时用它。注意控制顶层项数和层级深度——它最容易被塞成「深嵌套」。

顶部导航。横向空间有限,只放得下少数几个顶级入口。适合模块不多、想要横向并列的产品;后台如果模块很多,顶部就放不下,还是交给左侧菜单。

面包屑。不是用来「去别处」,而是用来告诉你「你现在在哪」、并提供「回上一层」的退路。一旦页面有了多级(列表 → 详情 → 子详情),面包屑几乎是必备,能极大缓解深层级带来的迷路感。

标签页(Tabs)。用在同一个对象的不同切面上:一个客户详情页,用「基本信息 / 订单记录 / 跟进日志」几个标签平铺,而不是做成三个菜单项或三层页面。它把「同一层、并列的几块内容」摊开,避免无谓的深钻。

一句话分工:左侧菜单管「在功能之间跳」,面包屑管「我在树的哪个位置」,标签页管「同一个东西的不同侧面」,顶部导航管「少数几个并列大区」。

4. 命名:一个功能,只用一个名字

导航能不能用,一半看命名。两条铁律:

① 一致性——同一个东西,处处叫一样。菜单里叫「客户」,按钮上就别叫「会员」,弹窗标题又写「用户」。同一个概念在不同地方换三个名字,用户会以为是三样东西,反复确认、反复犹豫。先定一套词,然后全站统一。

② 可预测、可区分。好标签让人没点开就猜得到点开是什么(可预测),并且和旁边的标签明显不同、不会混(可区分)。「数据管理」和「信息管理」并排放,用户根本分不清差别,只能两个都点开试——这就是命名没做到「可区分」。

5. 别让人迷路

层级一深,人就容易丢失方向感。三个低成本的「防迷路」措施,后台尤其值得做齐:

  • 当前位置高亮。左侧菜单里,用户正在看的那一项要明确高亮,所在的分组要自动展开。否则用户进了深页面,回头一看菜单全是收起来的,完全不知道自己在哪一支。
  • 面包屑随时在线。多级页面顶部放一条面包屑(首页 › 客户 › 张三 › 订单记录),既标明位置,又让每一层都能一键回去。
  • 合理的返回。从详情、弹窗、子页面出来,要有清楚的「返回 / 关闭」路径,别让用户只能靠浏览器后退键。给退路,就是给安全感。
⚠️ 这样的后台菜单,用户找东西像寻宝

• 顶层按数据库表分:「主实体」「关联实体」「字典表」「日志表」——没人看得懂。

• 全平铺成一长列三十多项,不分组,每次都得从头扫到尾。

• 或者反过来,嵌套五六层:「系统 › 业务 › 财务 › 报表 › 导出 › 导出报表」,点到怀疑人生。

• 同一功能多个名字:菜单「客户」、按钮「会员」、标题「用户」,以为是三个模块。

• 进了深层页面,菜单不高亮、没有面包屑,用户彻底不知道自己在哪、怎么回去。

✅ 这样的后台菜单,东西「猜得到在哪」

• 顶层按用户任务分,七八个以内的大分组:客户 / 订单 / 商品 / 报表 / 设置。

• 控制在 2~3 层:点开一个分组就能看到具体功能,不用层层下钻。

• 用户的词当标签,可预测、可区分,一个功能从头到尾只有一个名字。

• 当前位置高亮、所在分组自动展开;多级页面配面包屑,处处有退路。

🔑 后台菜单自查清单(做完一版菜单挨条过)
  1. 顶层每个分组,都对得上用户的一件任务吗?有没有按数据库表 / 部门来分的?
  2. 顶层项数一眼扫得完吗(大致七八个以内)?有没有摊成一长墙?
  3. 最常用的功能,点几次能到?有没有被埋到第四层、第五层?
  4. 每个菜单名,用户没点开能不能猜到里面是什么?是用户的词还是内部黑话?
  5. 同一个功能,在菜单 / 按钮 / 标题里是不是同一个名字?
  6. 相邻的两个标签,差别明显吗?会不会让人「两个都点开试」?
  7. 进到深层页面,菜单高亮了吗、分组展开了吗、有面包屑吗、能一键返回吗?

6. 试一试

✏️ 画一画你自己的后台菜单

拿你正在做(或最近用过)的一个后台,把左侧菜单的结构画成一棵树:顶层有哪些分组,每个分组下面挂哪些功能,挂到第几层。画完数三个数,再动手优化:

① 最深几层?从顶层点到最里面的功能,要点几下。超过 3 层的那几支,圈出来。
② 最宽几项?哪一层项数最多。顶层超过七八项、或某层排成一长列的,圈出来。
③ 有没有同物异名 / 黑话标签?把看不懂或叫法不统一的菜单名圈出来。

然后做三件事:

• 能合并的合并:意思相近、用户分不清的几项,并成一个,或收进同一个分组。
• 能提升的提升:埋太深的高频功能,往上挪一两层,让它两三步就能到。
• 能改名的改名:把黑话换成用户的词,把全站不统一的叫法统一成一个。

改完再回到上面的「实验台」对照一下:你的结构更像「太宽」「太深」还是「平衡」?往甜区靠一靠。

信息架构没有标准答案,但有个稳定的方向:顺着用户的心智分类、把层级压在 2~3 层、用对导航形式、再处处标明「你在哪」。做到这几条,用户就不用「学会」你的后台——他猜就能用对。这正是好交互最省力的样子。
