向导 · 弹窗 · 抽屉

🖼️ 容器选择 · 难度 ●●○ · 更新于 2026-06-30

后台系统里有三种最常用的"承载容器":向导、弹窗、抽屉。它们解决的问题不一样,却经常被混用——把一个简单确认框做成三步向导,把十几个字段的表单塞进一个小弹窗,或者用全页跳转来做本该是抽屉的事。选错了,用户操作的每一步都多摩擦:填到一半的表单不知道还有多少,关掉弹窗才发现背后列表的位置全丢了。这一页把三种容器说清楚:各自适合什么任务、为什么适合、选错的代价是什么。

1. Wizard 向导:带着用户走完复杂流程

向导把一个多步骤、线性的任务拆成一系列"页",每次只让用户面对一小块决策,配合顶部或侧边的进度指示器告诉他"现在在第几步,还有几步"。适合用向导的场景有几个共同特征:

  • 步骤之间有依赖关系:第一步不填完就没法进第二步,后面的选项依赖前面的输入。
  • 任务对用户比较陌生:一次性全展示会让人不知道从哪开始,逐步引导能降低认知负担。
  • 出错代价较高:初始化生产环境、提交正式申请这类操作,在最后一步加确认环节是必要的。
✅ 后台场景:适合用向导

• 新建员工档案:基本信息 → 合同与薪资 → 分配角色权限 → 确认提交。步骤之间有逻辑依赖,单独给一张大表单会让人不知道先填哪里。

• 初始化新项目:选部署环境 → 填写域名配置 → 绑定数据库 → 预览并启动。每步的输入决定下一步的选项。

• 首次系统配置:安装后引导管理员设置组织信息、创建第一个账号、选择初始模块——用户只操作一次,陌生感最强,向导最合适。

❌ 什么时候不要用向导

任务不复杂、用户已经很熟悉、或只需填 2-3 个字段——这些情况用向导是过度包装。用户明明只是点"确认删除",却被迫走"选择原因 → 确认身份 → 点击完成" 3 步,每次都是折磨。向导是给陌生复杂任务准备的,不是给所有操作加仪式感用的。

2. Modal 弹窗:聚焦、短、必须打断

弹窗在页面中央弹出,用半透明遮罩把背景压暗,强制用户先处理这件事。这是它的优点,也是它的边界:既然要打断,就要值得打断。弹窗适合的任务有三个特征:操作短(几秒内完成)、需要用户聚焦(不需要参考背景内容)、必须先于其他操作完成。

✅ 后台场景:适合用弹窗

• 删除确认:"确认删除这条客户记录?删除后不可恢复。"这是弹窗的主场:操作不可逆、只有两个按钮、用不着看背景。

• 快速填写单字段:在列表里直接修改某条记录的备注或标签——弹窗一个输入框加确认就够了,不需要跳整页。

• 强制提示 / 授权:许可证即将过期,需要用户确认知晓后才能继续——强制打断是合理的,这正是遮罩的意义。

❌ 弹窗的两大死亡陷阱

把复杂表单塞进弹窗:字段一多就出现内部滚动条,用户填到一半不知道还有多少。移动端键盘弹出来还会遮住弹窗下半部分。字段超过 5-6 个就该考虑抽屉或全页了。

弹窗套弹窗:弹窗里有个"选择客户"按钮,点开又弹一个选择器。用户点到第三层已经不知道自己在哪。弹窗最多一层,有嵌套选择需求就用页内选择器或抽屉里的内嵌组件。

3. Drawer 抽屉:保留上下文、中等体量

抽屉从页面右侧滑入,背后的页面依然可见但被轻度压暗。这个设计有明确含义:用户处理眼前这件事时,背景的上下文还在。这是弹窗做不到的——弹窗的遮罩设计意味着"先把背景全忘了"。

抽屉的优势就在"边看边做"这个场景:用户在列表里看到一条记录,想编辑它,但又需要随时瞥一眼其他行来对照——这正是抽屉的主场。体量上,抽屉比弹窗能放更多字段(7-15 个都还舒服),比全页跳转代价低(不打断列表的位置和筛选状态)。

✅ 后台场景:适合用抽屉

• 记录详情 / 编辑:客户列表里点某行,右侧抽屉展开该客户的完整信息,直接在抽屉里编辑并保存,列表不刷新,用户保留了在哪一行的感知。

• 筛选 / 过滤面板:点击"筛选"按钮,从右侧滑入一个包含多个过滤条件的面板,调整条件后点"应用",背后列表实时刷新——用户全程能看到列表在变化。

• 新建(中等复杂度):需要填 8-12 个字段的"新建客户",不够复杂到要拆多步向导,但塞进弹窗又太挤——右侧抽屉正合适。

❌ 什么时候不要用抽屉

任务完全不需要看背景内容,或者步骤特别多——抽屉没有进度指示器,用户会迷失。移动端屏幕窄,右侧抽屉可能几乎占满全屏,这时候还不如直接跳全页或用底部弹出层。

🔑 三选一决策表
判断项 向导 弹窗 抽屉
任务有几步? 3步以上,有依赖 1步,即问即答 1步,字段较多
用户熟悉这个操作吗? 陌生,需要引导 熟悉,不需要引导 熟悉,直接上手
需要看背后的上下文吗? 不需要(专注完成) 不需要(快速确认) 需要(边看边填)

4. 选错容器的代价

容器选错不是"设计风格偏差",它直接影响用户完成任务的速度和出错率:

  • 简单确认做成向导:用户想删一条记录,结果要走"选择操作类型 → 填写删除原因 → 二次确认 → 提交" 4 步。每次删除都是一次小手术,用户慢慢学会根本不删,让垃圾数据堆积。
  • 十个字段塞进弹窗:弹窗出现内部滚动,第 7 个字段要手动滚才能看见,按钮可能被键盘顶出视口。填完表单有相当比例的用户会误触关闭按钮,数据全没。
  • 边看边填的场景用全页跳转:用户在"采购列表"里找到一行要编辑,点开全页,填完回来已经丢失了列表的滚动位置和筛选状态,又得重新找。
  • 多步骤操作用抽屉:抽屉没有步骤感,用户在"第几步"完全靠自己记忆,容易跳步或遗漏必填项,出错了也不知道该改哪里。

5. 试一试

✏️ 给你后台的三个操作各选一种容器

挑你正在做或熟悉的一个后台系统(NocoBase 的某个模块也行),找三个操作,分别决定用向导、弹窗还是抽屉。不用每种各选一次,允许重复——关键是能说出理由。

操作 1：____________ → 用 ___ 理由：____________
操作 2：____________ → 用 ___ 理由：____________
操作 3：____________ → 用 ___ 理由：____________

判断时问自己三个问题:

  1. 这个操作有几步?步骤之间有依赖吗?（多步有依赖 → 向导）
  2. 用户需要看着背景内容来填吗?（需要上下文 → 抽屉）
  3. 任务短、需要打断、或者是确认类?（短且打断 → 弹窗）

没有唯一答案——同一个操作在不同系统里可以有不同的合理选择。写出理由比选对答案更重要。

三种容器不是风格偏好,是针对不同任务复杂度和上下文需求的工具。用对了,用户顺手到感觉不到容器的存在;用错了,每一步都是额外的摩擦。下次设计一个新功能入口时,先问自己:它有几步?用户需要参考背景吗?操作本身复杂还是简单?——这三个问题基本就能锁定正确的容器。
